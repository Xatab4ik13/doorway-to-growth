
## Что делаем

Подтянуть с `core.brandoors.ru` для всех 7 коллекций (PRIME, GHOST, HEAVY, REFLECT, MAZE, ESTETICA, Входные двери):

1. **Описание** модели (`products.description`)
2. **Характеристики** (тип, стиль, материал, покрытие, толщина) — в `specifications.characteristics`
3. **Все опции SKU** (цвета, остекление, кромка/молдинг, наличник, панели) — в `specifications.*` и в `product_images.{variant_key,glazing_key,molding_key}`
4. **Размерные сетки** ширины и высоты на модель — в `specifications.{widths,heights}`
5. **Структура конфигуратора** на странице товара — те же шаги что у brandoors.ru

Цены и формулы расчёта **не трогаем** (по решению).

## Структура конфигуратора (как на brandoors.ru)

Общие шаги для всех коллекций:
- Ширина, мм — чипы (400…1000+)
- Высота, мм — чипы (1500…2500+)
- Цвет — свотчи

Доп. оси по коллекциям:
- **PRIME** — Остекленение
- **HEAVY** — Кромка
- **MAZE** — Наличник
- **REFLECT** — Внешняя панель + Внутренняя панель + Кромка
- **GHOST** — без доп. осей (только «Под покраску»)
- **ESTETICA** — только цвет
- **Входные двери** — Внутренняя панель + Цвет внутренней панели

Под конфигуратором: **Описание**, **Характеристики** (key→value), блоки ссылок **Погонаж** и **Фурнитура** на соответствующие категории.

## Технический план

### Шаг 1. Скрапер источника
Скрипт обходит все продукты в коллекциях источника, собирает per‑product:
- `description`, `characteristics`
- все SKU c `propertyValues` → группирует по property.slug
- `productSizes` (нужна доп. query) → списки ширин и высот

Результат — `/tmp/source_all.json`.

### Шаг 2. Миграция данных (один большой supabase--insert)
Для каждого товара в БД сделать `UPDATE products SET description=…, specifications = specifications || jsonb_build_object('characteristics', …, 'widths', …, 'heights', …, 'axes', …)`.

Для тех моделей, где сейчас в БД нет вариантных фото с новыми ключами (color/glazing/molding/inner_panel/outer_panel), добавить недостающие строки `product_images` с `variant_key`/`glazing_key`/`molding_key` (расширять схему пока не будем — 3-х осей хватает для всего кроме REFLECT, где есть 4 оси: для REFLECT уложим Внутреннюю+Внешнюю панели в `glazing_key` склейкой `"InnerKey | OuterKey"`).

Зеркало SQL — в `infra/migrations/2026…_doors_metadata.sql`.

### Шаг 3. UI конфигуратора (`src/pages/StorefrontProduct.tsx`)
- Добавить чипы **Ширина** и **Высота** перед существующими селекторами цвета/остекления/молдинга. Источник списков — `product.specifications.widths/heights`.
- Подписи селекторов берём из `specifications.axes` (по коллекции). Например для MAZE подпись для molding-оси будет «Наличник», а не «Молдинг».
- Под конфигуратором — секция **Характеристики**: рендер `specifications.characteristics` как «Ключ: Значение».
- Под характеристиками — два блока ссылок:
  - **Погонаж** → ссылки на категории Наличники / Плинтус / Короб / Добор / Притворная планка
  - **Фурнитура** → ссылки на категории фурнитуры
- Размер сохраняется в строке корзины (`cartItem.options.width/height`).

### Шаг 4. Тип `product_images` и хук
`molding_key` и `glazing_key` уже есть. Никаких миграций схемы.
В `useStorefrontData.ts` подтянуть `description` и `specifications` (уже подтягиваются).

### Шаг 5. Проверка
Playwright по 5 страницам (по одной из PRIME/HEAVY/MAZE/REFLECT/GHOST) — скриншоты, что:
- видны чипы Ширина/Высота с правильными значениями,
- видны нужные оси с правильными подписями,
- ниже — Описание и Характеристики,
- ниже — ссылки Погонаж/Фурнитура,
- при смене опций корректно меняется фото (где есть фото).

## Что НЕ делаем

- Цены, надбавки, расчёт стоимости — без изменений.
- Дизайн конфигуратора не меняем, только добавляем новые поля в существующий layout.
- Не добавляем новые таблицы и не меняем схему БД.
